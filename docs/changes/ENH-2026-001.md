## Problem

The current TR system allows a Transport Request to be created and moved directly into DEVELOPMENT status with no formal document gate. There is no structured intake process — no problem statement, no solution definition, no scope boundaries, and no test scenarios required before development begins. This means:

- Code can be written without a documented reason
- Scope creep has no enforcement mechanism
- Claude Code has no structured work order to build from
- There is no audit trail of WHY a change was requested before it was built

---

## Solution

Introduce a formal CR/ENH intake gate into the Nerve Center Transport Wing. Every TR must now pass through DRAFT and APPROVED stages before development can begin. The TR creation form becomes a structured document with mandatory fields. An Approve button moves it to APPROVED. A Copy Work Order button generates a machine-readable spec for Claude Code. No development starts and no code is pushed until a TR exists in APPROVED status.

---

## In Scope

- Add `DRAFT` and `APPROVED` to the TR status enum in Prisma schema
- Add `problem`, `solution`, `inScope`, `outOfScope` text fields to `TransportRequest` model
- Add `TRTestScenario` model (id, trId, title, steps, expectedResult, result, testedBy, testedAt, notes)
- Update TR create endpoint — default status is now `DRAFT` (not `DEVELOPMENT`)
- New `PATCH /api/platform/transport/:id/approve` endpoint — moves DRAFT → APPROVED
- Promote endpoint blocks unless TR is in `APPROVED` status when moving to `DEVELOPMENT`
- Update `TransportNew.jsx` — add Problem, Solution, In Scope, Out of Scope fields to the form
- Update `TransportDetail.jsx` — add Approve button (visible on DRAFT TRs), add Copy Work Order button, add Test Scenarios section
- Update `TransportManager.jsx` — add DRAFT and APPROVED columns to the Kanban board
- Run `prisma migrate dev` to apply schema changes

---

## Out of Scope

- No changes to the Promote → Quality or Promote → Production flow
- No changes to Rollback functionality
- No WhatsApp notifications for DRAFT/APPROVED transitions (Phase 5 feature)
- No auto-detection of git pushes (developer still pushes manually)
- No CI/CD pipeline integration
- No changes to any tenant-facing modules

---

## Test Scenarios

1. **DRAFT gate enforced** — Create a new TR via the form. Verify TR is created with status `DRAFT`, not `DEVELOPMENT`. Verify the Promote button is absent on a DRAFT TR.

2. **Approve flow** — Open a DRAFT TR in TR Detail. Click Approve. Verify status changes to `APPROVED`. Verify audit log records the approval action with timestamp and approver name.

3. **Work Order generation** — Open an APPROVED TR with all fields filled (problem, solution, in scope, out of scope, test scenarios). Click Copy Work Order. Verify clipboard contains a correctly formatted work order with all TR fields populated.

4. **Development gate** — Attempt to move an APPROVED TR to DEVELOPMENT (begin work). Verify it transitions correctly. Verify a TR still in DRAFT cannot be moved to DEVELOPMENT.

5. **Kanban board** — Verify DRAFT and APPROVED columns appear on the Transport Manager Kanban board and TR cards display correctly in each column.

## Notes

- This is a bootstrapping change — it is being built before the formal intake system exists, so ENH-2026-001 itself serves as the first document under the new process
- After this is live, every future feature must have a CR/ENH document in `docs/changes/` before development begins
- Document ID format: `CR-YYYY-NNN` for change requests, `ENH-YYYY-NNN` for enhancements
