## Problem

The current TR system has four issues that make it unsafe for production use:

**Issue 1 — No environment labels on Kanban**
The Kanban columns show status names (DEVELOPMENT, TESTING, IN_QUALITY, IN_PRODUCTION) but do not display which Git branch they map to. Developers cannot tell at a glance which environment a TR is currently deployed in.

**Issue 2 — Promote button has no test gate**
Any TR in TESTING or IN_QUALITY can be promoted regardless of whether test scenarios have been run or passed. This means untested code can be pushed to Quality and then to Production with a single click.

**Issue 3 — No DEV vs QUALITY test separation**
There is only one pool of test scenarios per TR. There is no distinction between DEV-level testing (does the feature work?) and Quality-level testing (does it break anything existing?). Both environments need their own test layers.

**Issue 4 — Production has a promote button**
IN_PRODUCTION TRs show a promote button which has no valid target. Production is the final destination. The only valid action there is Rollback. Having a promote button on production TRs is confusing and potentially dangerous.

---

## Solution

Fix all four issues to turn the TR Kanban into a proper three-environment release pipeline with enforced testing gates at each promotion point.

---

## In Scope

### Fix 1 — Environment Labels on Kanban Board
- Redesign `TransportManager.jsx` Kanban to show three environment blocks instead of four individual columns:
  - **DEV** block (branch: dev) — contains DEVELOPMENT + TESTING columns side by side
  - **QUALITY** block (branch: quality) — contains IN_QUALITY column
  - **PRODUCTION** block (branch: main) — contains IN_PRODUCTION column
- Each block has a header showing environment name + branch name
- Visual separator between the three blocks

### Fix 2 — Test Gate on Promote Button
- Promote button from TESTING → IN_QUALITY is blocked unless:
  - At least 1 DEV test scenario exists on the TR
  - All DEV test scenarios have result = PASSED (none PENDING or FAILED)
- Promote button from IN_QUALITY → IN_PRODUCTION is blocked unless:
  - At least 1 QUALITY test scenario exists on the TR
  - All QUALITY test scenarios have result = PASSED (none PENDING or FAILED)
- When blocked: button is disabled + tooltip shows exact reason (e.g. "2 test scenarios pending")
- Gate enforced on both TransportManager card and TransportDetail page

### Fix 3 — DEV vs QUALITY Test Scenario Layers
- Add `layer` field to `TransportTestScenario` model: `DEV` or `QUALITY`
- Default layer = `DEV` for backward compatibility with existing scenarios
- TransportDetail Tests tab shows two sections:
  - **DEV Test Scenarios** — run during TESTING stage
  - **QUALITY Test Scenarios** — run during IN_QUALITY stage
- Add Scenario button respects current TR status:
  - If TESTING → adds to DEV layer
  - If IN_QUALITY → adds to QUALITY layer
- Gate check uses correct layer for each promotion:
  - TESTING → IN_QUALITY: checks DEV layer scenarios
  - IN_QUALITY → IN_PRODUCTION: checks QUALITY layer scenarios

### Fix 4 — Production Lock (No Push Button)
- Remove promote button from IN_PRODUCTION TRs entirely
- IN_PRODUCTION TR actions: Rollback only
- Add "LIVE" badge on IN_PRODUCTION cards in Kanban
- Production column header shows a lock icon
- TransportDetail shows a "This TR is in Production" banner with rollback as the only action

### Fix 5 — Test Scenario Auto-Import from Changes Made File
- When creating a TR, if a Changes Made file is uploaded:
  - Parse the file for `## TEST SCENARIOS (DEV)` and `## TEST SCENARIOS (QUALITY)` sections
  - Auto-create `TransportTestScenario` records from each numbered item found
  - DEV scenarios imported with layer = DEV
  - QUALITY scenarios imported with layer = QUALITY
- If no scenarios section found in file → no auto-import, manual add still works
- Show import summary after TR creation: "X DEV + Y QUALITY test scenarios imported"

## Test Scenarios

1. **Environment labels visible** — Open Transport Manager. Verify three environment blocks are visible: DEV (branch: dev), QUALITY (branch: quality), PRODUCTION (branch: main). Verify DEVELOPMENT and TESTING columns appear inside the DEV block.

2. **Test gate blocks promotion — TESTING** — Create a TR with no test scenarios. Move it to TESTING. Verify Promote button is disabled with message "Add at least one test scenario before promoting." Add a scenario and mark it FAILED. Verify promote still blocked. Mark it PASSED. Verify promote button activates.

3. **Test gate blocks promotion — IN_QUALITY** — Promote a TR to IN_QUALITY. Verify the QUALITY test scenarios section appears. Verify promote to Production is blocked until at least one QUALITY scenario is added and marked PASSED.

4. **DEV vs QUALITY scenario separation** — Open a TR in TESTING. Add a scenario — verify it is tagged DEV. Promote to IN_QUALITY. Add a scenario — verify it is tagged QUALITY. On the Tests tab, verify two separate sections: DEV Test Scenarios and QUALITY Test Scenarios.

5. **Production lock** — Open an IN_PRODUCTION TR. Verify there is no Promote button. Verify only Rollback button is present. Verify a "LIVE" badge appears on the card in Kanban.

6. **Auto-import test scenarios** — Create a TR and upload a Changes Made file that contains `## TEST SCENARIOS (DEV)` and `## TEST SCENARIOS (QUALITY)` sections. After TR creation, verify the scenarios were auto-imported with correct layers and appear in the Tests tab.
