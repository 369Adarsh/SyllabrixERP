## Problem

The current TR promotion flow moves a TR directly from TESTING → IN_QUALITY and from IN_QUALITY → IN_PRODUCTION in a single step with no receipt confirmation. This means:

1. There is no distinction between a TR arriving in an environment (code merged) and a TR being actively implemented/accepted in that environment
2. Quality testing begins immediately on promotion with no admin acknowledgement
3. There is no way for admins to batch-receive multiple TRs before choosing when to implement them
4. The DEV page shows DEV, Quality, and Production all mixed — developer cannot focus on just their environment
5. After pushing to Quality, there is no confirmation message telling the developer to verify receipt in the Quality environment

---

## Solution

Introduce a two-step Receipt → Implement flow for both Quality and Production environments. When a TR is promoted, it first lands in a RECEIVED state. The admin then either implements it manually (by clicking Implement) or has it auto-implemented if the auto-implement setting is enabled. After implementation, the environment's test scenarios become active (Quality) or the TR goes live (Production).

## In Scope

### New TR Statuses
- Add `IN_QUALITY_RECEIVED` — TR has been promoted to quality branch but not yet implemented
- Add `IN_PRODUCTION_RECEIVED` — TR has been promoted to production branch but not yet implemented
- Updated status flow:
  ```
  DRAFT → APPROVED → DEVELOPMENT → TESTING
       → IN_QUALITY_RECEIVED → IN_QUALITY (testing active)
       → IN_PRODUCTION_RECEIVED → IN_PRODUCTION (live)
  ```

### New TRLogAction values
- `RECEIVED_IN_QUALITY` — logged when TR arrives in quality
- `IMPLEMENTED_IN_QUALITY` — logged when TR is implemented in quality
- `RECEIVED_IN_PRODUCTION` — logged when TR arrives in production
- `IMPLEMENTED_IN_PRODUCTION` — logged when TR is implemented in production

### Prisma Schema
- Add `IN_QUALITY_RECEIVED` and `IN_PRODUCTION_RECEIVED` to `TRStatus` enum
- Add `RECEIVED_IN_QUALITY`, `IMPLEMENTED_IN_QUALITY`, `RECEIVED_IN_PRODUCTION`, `IMPLEMENTED_IN_PRODUCTION` to `TRLogAction` enum
- Add `autoImplementQuality Boolean @default(false)` to `TRSettings` model
- Add `autoImplementProduction Boolean @default(false)` to `TRSettings` model
- Create `TRSettings` model if it does not exist (one record per platform)

### Backend Changes
- Update `promote()` in transport.service.js:
  - TESTING → `IN_QUALITY_RECEIVED` (GitHub API merges dev→quality)
  - After merge: if `TRSettings.autoImplementQuality === true` → immediately move to `IN_QUALITY` and log both actions
  - `IN_QUALITY` → `IN_PRODUCTION_RECEIVED` (GitHub API merges quality→main)
  - After merge: if `TRSettings.autoImplementProduction === true` → immediately move to `IN_PRODUCTION` and log both actions
- New `implement(id, adminName)` service function:
  - If status is `IN_QUALITY_RECEIVED` → move to `IN_QUALITY`, log `IMPLEMENTED_IN_QUALITY`
  - If status is `IN_PRODUCTION_RECEIVED` → move to `IN_PRODUCTION`, log `IMPLEMENTED_IN_PRODUCTION`
  - Any other status → throw error "TR is not in a received state"
- New `getSettings()` and `updateSettings()` service functions
- New `POST /:id/implement` route and controller handler
- New `GET /settings` and `PATCH /settings` routes and handlers

### Frontend — TransportManager.jsx
- **DEV tab:** After clicking Push to Quality, show success banner: "TR pushed to Quality — switch to the Quality tab to confirm receipt"
- **Quality tab:** Show `IN_QUALITY_RECEIVED` TRs with "RECEIVED" badge and **"Implement"** button (teal). Show `IN_QUALITY` TRs with QUALITY test scenarios and push to Production button
- **Production tab:** Show `IN_PRODUCTION_RECEIVED` TRs with "RECEIVED" badge and **"Implement"** button. Show `IN_PRODUCTION` TRs with LIVE badge and Rollback only
- Update STATUS_META for new statuses
- Update NEXT_LABEL and promote logic for new flow

### Frontend — Transport Settings Page
- Add a Settings tab or section on the Transport Manager page
- Toggle: **Auto-implement in Quality** (on/off) — if on, TR moves from RECEIVED → IN_QUALITY automatically
- Toggle: **Auto-implement in Production** (on/off) — if on, TR moves from RECEIVED → IN_PRODUCTION automatically
- Save settings via PATCH /api/platform/transport/settings
- Load settings on page mount and display current state

### Frontend — TransportDetail.jsx
- Update STATUS_META, LOG_ICON, NEXT_LABEL for new statuses
- Show "Implement" button on RECEIVED TRs
- Show correct action buttons per status:
  - `IN_QUALITY_RECEIVED`: Implement button only
  - `IN_QUALITY`: QUALITY test scenarios + Push to Production (if all QUALITY tests pass)
  - `IN_PRODUCTION_RECEIVED`: Implement button only
  - `IN_PRODUCTION`: Rollback only, LIVE badge

### Frontend — platform.js API
- Add `implementTR(id)` — POST /transport/:id/implement
- Add `getTRSettings()` — GET /transport/settings
- Add `updateTRSettings(data)` — PATCH /transport/settings

## Test Scenarios

1. **DEV to Quality promotion shows receipt state** — Create a TR, move to TESTING, pass all DEV scenarios, click Push to Quality. Verify TR status becomes `IN_QUALITY_RECEIVED`. Verify DEV page shows success banner. Switch to Quality tab — verify TR appears with RECEIVED badge and Implement button (not yet in testing).

2. **Manual implement in Quality** — With auto-implement OFF: Open received TR in Quality tab, click Implement. Verify status moves to `IN_QUALITY`. Verify QUALITY test scenario section becomes active. Verify audit log records `RECEIVED_IN_QUALITY` and `IMPLEMENTED_IN_QUALITY` as separate events.

3. **Auto-implement in Quality** — Enable auto-implement Quality in Settings. Promote a TESTING TR. Verify it skips `IN_QUALITY_RECEIVED` and goes directly to `IN_QUALITY`. Verify both log entries still created.

4. **Quality to Production promotion shows receipt state** — Pass all QUALITY test scenarios on an IN_QUALITY TR. Click Push to Production. Verify status becomes `IN_PRODUCTION_RECEIVED`. Switch to Production tab — verify TR appears with RECEIVED badge and Implement button.

5. **Manual implement in Production** — With auto-implement OFF: Click Implement on received Production TR. Verify status moves to `IN_PRODUCTION`. Verify LIVE badge appears. Verify only Rollback button is available — no Implement or Promote.

6. **Settings toggle persists** — Enable auto-implement Quality, disable auto-implement Production. Refresh page. Verify settings are preserved correctly.

## Notes

- Existing TRs in `IN_QUALITY` or `IN_PRODUCTION` remain unaffected — they retain current status
- The `TRSettings` table holds one platform-wide record — created on first access if it does not exist
- The implement action is logged as two separate audit entries (RECEIVED + IMPLEMENTED) to maintain full traceability of when code arrived vs when it was activated
