## 1. Syllabrix Universal ID System

### Format

```
SYL + 5-digit zero-padded sequential number
```

| Example | Entity |
|---|---|
| `SYL00001` | First registered tenant |
| `SYL00023` | Sharma Grocery Chain (tenant) |
| `SYL00028` | Main Market branch (Sharma Chain) |

### Key rules

- **Single global pool** — tenants, branches, and users draw from the same counter. The ID does not encode entity type, business category, or branch/tenant/human status. It is a pure opaque unique token.
- **Sequential, not random** — assigned in order of creation (oldest entity gets the lowest number).
- **Permanent** — once assigned, an ID never changes, is never recycled, and is never reused.
- **DB-enforced uniqueness** — `syllabrixId` carries `@unique` on both `Tenant` and `Branch` models in the Prisma schema.
- **Never hand-craft an ID** — always call `nextSyllabrixId()`. Never write `SYL00099` or similar in seed scripts.

### Generator

| Item | Detail |
|---|---|
| File | `backend/src/utils/syllabrixId.js` |
| Export | `nextSyllabrixId()` — async, returns the next `SYLnnnnn` string |
| Logic | Reads the current max numeric suffix across `tenants`, `branches`, and `users` tables, returns `max + 1` padded to 5 digits |
| Auto-wired | Tenant registration → `auth.service.js` · Branch creation → `branches.service.js` · User creation → `users.service.js` |
| Batch | `nextNSyllabrixIds(n)` — single DB read, returns n consecutive IDs. Use when creating tenant + user together |

### Current allocation (as of 2026-05-25)

| Range | Count | Entity type |
|---|---|---|
| `SYL00001` – `SYL00024` | 24 | Tenants (all registered businesses) |
| `SYL00029` – `SYL00056` | ~28 | Owner/Admin users (business-level SYL IDs) |
| **`SYL00062`** | — | **Next available for tenants + business-level owners** |

Branches no longer use the SYL sequential pool — they use `SYL00023-MAIN` format (see Branch Pattern).
Branch-assigned staff no longer use the SYL sequential pool — they use `MAIN1001` format (see Section 4).

Notable assignments:
- `SYL00005` — Sharma Kirana Store *(tenant)*
- `SYL00022` — Iron Zone Fitness *(tenant)*
- `SYL00023` — Sharma Grocery Chain *(parent tenant)*
- `SYL00025` — Station Road branch
- `SYL00026` — Palasia Square branch
- `SYL00027` — Vijay Nagar branch
- `SYL00028` — Main Market branch (HQ)
- `SYL00051` — Iron Zone owner user
- `SYL00056` — Raj, Owner, Sharma Grocery Chain *(user)*
- `SYL00057`–`SYL00060` — Four Sharma Chain branch managers *(users)*

### Branch pattern

When a business opts into multi-branch (`hasBranches: true`):
- The **parent tenant keeps its original ID** — no change.
- Each branch gets an ID of the form **`SYL00023-MAIN`** — the tenant's SYL ID + dash + the branch's 4-char uppercase alpha code.
- This makes ownership immediately visible: `SYL00023-STRD` is unambiguously a branch of tenant `SYL00023`.
- Generated by `generateBranchId(tenantSyllabrixId, branchCode)` in `syllabrixId.js`.

### Legacy formats (all migrated — never use again)

| Old format | Was used for |
|---|---|
| `SYL` + 6 random alphanumeric (e.g. `SYLS7Q876`) | Regular tenant IDs (pre-2026-05-25) |
| `SYLB2B001` – `SYLB2B006` | B2B marketplace tenants |
| `SYLBR-{NAME}-{CODE}` (e.g. `SYLBR-SHARMA-MAIN`) | Branch IDs |

## 2. Branch RBAC Architecture

### Access model

| Role | Scope | How enforced |
|---|---|---|
| **Owner / Admin** | All branches + tenant-level | Sidebar branch switcher; `null` branchId = all-branches view |
| **Branch Manager** | Their branch only | `branchId` stamped on JWT at login; `injectBranch(req)` in controllers |
| **Branch Staff** | Their branch only | Same as manager |

### injectBranch pattern

Every controller that handles branch-scoped data must use this helper:

```js
const injectBranch = (req) => {
  const params = { ...req.query };
  if (req.user?.role === 'MANAGER' && req.user.branchId) params.branchId = req.user.branchId;
  return params;
};
```

Applied to: expenses, attendance, assets, returns, quotations, purchase orders, GRNs, bills, inventory, reports.

### useBranch pattern (frontend)

`BranchContext` → `useBranch()` hook:
- For **MANAGER**: reads `user.branchId` from JWT (fixed — cannot change).
- For **OWNER**: reads from `localStorage` (`syl_br_${tenant.id}`); `null` = All Branches view.

Every list/load call on a branch-scoped page must pass `branchId ? { branchId } : {}` as params. Every create call must stamp `...(branchId && { branchId })` on the payload.

### Manager scope rule

A branch manager controls roles and staff within their own branch only. The owner controls everything across all branches. A manager can never override their `branchId` by passing a different one in the request — the backend always overwrites it from the JWT.

## 4. Human (User) ID Standard

**Issued:** v1.1, 2026-05-25 (updated from v1.0)  
**Mandatory:** Every user — owner, manager, employee, and staff — **must** have a `syllabrixId`. No user goes live without one.

### Two ID types based on role

| User type | ID format | Example | Generator |
|---|---|---|---|
| Owner / Admin (business-level) | `SYL00056` — global SYL sequential pool | `SYL00056` | `nextSyllabrixId()` |
| Manager / Staff (branch-assigned) | `XXXX1001` — 4-char branch code + 4-digit non-zero-leading sequential | `MAIN1001`, `STRD1002` | `generateStaffId(branchCode)` |

### Staff ID rules

- **8 characters total** — 4 uppercase alpha (branch code) + 4 numeric digits.
- **Numeric part starts at 1001** — never has a leading zero; always 4 digits (range 1001–9999).
- **Branch-scoped but globally unique** — `@unique` on `User.syllabrixId`; generator scans ALL users globally to find the max for a given prefix.
- **Readable ownership** — `MAIN1001` immediately signals "employee at the MAIN branch".

| Example | Who |
|---|---|
| `SYL00056` | Raj — Owner of Sharma Grocery Chain (business-level) |
| `MAIN1001` | Amit Sharma — Manager, Main Market branch |
| `STRD1001` | Mohan Lal — Manager, Station Road branch |
| `PLSA1001` | Neha Jain — Manager, Palasia Square branch |

### Generators (all in `backend/src/utils/syllabrixId.js`)

| Function | Used for |
|---|---|
| `nextSyllabrixId()` | Single business-level user |
| `nextNSyllabrixIds(n)` | Batch — tenant + owner at registration (atomic, no race) |
| `generateStaffId(branchCode)` | Branch-assigned manager or staff |

### Card / Surface Rules

- **No shadows** — all surfaces use `border: 1px solid var(--border)` only. No `box-shadow` on page-level cards.
- **Card radius** = `var(--radius-lg)` = 12px.
- **Table wrapper radius** = `var(--radius-xl)` = 14px.
- **`Card` component** (`frontend/src/components/ui/Card.jsx`) enforces this — never add inline `boxShadow`.

### Input / Button Rules

- All inputs: `border: 1.5px solid var(--border)`, `border-radius: var(--radius-md)` = 9px.
- All buttons: `border-radius: var(--radius-md)` = 9px.
- Primary CTA: `var(--navy)` background with white text, or `var(--cyan)` for teal CTAs.

### Page Structure Rules (all business pages)

Every business page must import `{ P } from '../../styles/page'` and use:
- `P.wrap(isMobile)` — outermost div
- `P.head` — header row (title + actions)
- `P.h1(isMobile)` — page title h1
- `P.sub` — subtitle/description
- `P.tableWrap` + `P.tableScroll` — table container
- `P.table`, `P.thead`, `P.th()`, `P.td()`, `P.tr(i, total)` — table internals

### Do NOT

- Do NOT use `boxShadow` on page-level cards.
- Do NOT hardcode radius values in px — always use CSS variables or `P.*` tokens.
- Do NOT create page-local `const INPUT` or `const CARD` style objects — use `P.*` tokens.
- Do NOT use pixel font sizes for h1 directly — use `P.h1(isMobile)`.
- Do NOT use amber (`#D97706`) as a brand or accent color — semantic warnings only.
- Do NOT use the retired colors `#312E81`, `#FFFBF0`, or `#E7E5E4` in new code.
- Do NOT use `--electric` for anything other than the AI Copilot / special feature nav link.

## 7. Document ID Standards

**Issued:** v1.0, 2026-05-25  
**Mandatory:** All business documents must carry a Syllabrix-standard document number.

### Format

```
SYL{TYPE}-{YYMM}-{NNNNN}
```

| Part | Meaning |
|---|---|
| `SYL` | Syllabrix namespace prefix |
| `{TYPE}` | Three-to-six letter type code (see table) |
| `{YYMM}` | Two-digit year + two-digit month (e.g. `2605` = May 2026) |
| `{NNNNN}` | Five-digit zero-padded monthly sequence |

### Document type codes

| Code | Document type | Example | Generator |
|---|---|---|---|
| `SYLINV` | Sales invoice | `SYLINV-2605-00001` | `generateInvoiceNumber()` in `backend/src/utils/generateNumber.js` |
| `SYLBIL` | Vendor bill | `SYLBIL-2605-00001` | `generateBillNumber()` in `backend/src/modules/bills/bills.service.js` |
| `RCP` | Payment receipt | `RCP-2605-00001` | `generateReceiptNumber()` in `backend/src/utils/generateNumber.js` |
| `FEE` | Fee receipt | `FEE-2605-00001` | `generateFeeReceiptNumber()` in `backend/src/utils/generateNumber.js` |
| `PO` | Purchase order | `PO-2605-00001` | `generatePONumber()` in `backend/src/utils/generateNumber.js` |

### Key rules

- **Monthly sequence resets** — the 5-digit counter restarts at `00001` each calendar month.
- **Globally unique within type** — uniqueness is enforced by `startsWith(prefix)` lookup across all tenants before inserting.
- **Forward-only** — existing pre-standard numbers are not retroactively renamed; new documents always use the current format.
- **Never hand-craft** — always call the relevant generator function.
- When adding a new document type: add a row to the table above, pick a `SYL{TYPE}` code that does not conflict, and implement the generator following the existing pattern.
